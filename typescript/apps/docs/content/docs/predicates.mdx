| Need | Builder |
|---|---|
| Email, URL, phone, UUID | `w.string().email()`, `.url()`, `.phone()`, `.uuid()` |
| SSN, EIN, ITIN, TIN | `w.string().ssn()`, `.ein()`, `.itin()`, `.tin()` |
| Financial identifiers | `w.string().creditCard()`, `.iban()`, `.abaRouting()`, `.cusip()` |
| Healthcare and reference codes | `w.string().npi()`, `.icd10Code()`, `.usState()`, `.currencyCode()` |
| Cross-field business rules | `w.object(...).when(...)`, `.requireOneOf(...)`, `.requireSum(...)` |

## Predicate IR

Predicates are the boolean evaluation units inside constraints. A constraint pairs a predicate with error metadata:

```json
{
  "pred": { "type": "call", "name": "is_email" },
  "error": {
    "code": "INVALID_EMAIL",
    "message": "Enter a valid email address"
  }
}
```

Work with predicate IR directly for advanced rules, schema tooling, and tests.

## Predicate Types

| Type | Description |
|------|-------------|
| `true` / `false` | Constant predicates |
| `regex` | Regular expression match |
| `template_literal` | Structured string scanner made from literal and character-class parts |
| `min_len` / `max_len` | String or array length |
| `range` | Numeric range |
| `exists` | JSON Pointer exists and is not null or undefined |
| `eq` | Value at path equals a literal |
| `in` | Value at path is one of several values |
| `required_with` | A field is required when another field exists |
| `required_without` | A field is required when another field is absent |
| `exactly_one_of` | Exactly one path in a set exists |
| `eq_fields` | Two fields are equal |
| `gt_field` / `gte_field` | Field comparison, greater than |
| `lt_field` / `lte_field` | Field comparison, less than |
| `sum_equals` | Sum of fields equals another field |
| `sum_equals_value` | Sum of fields equals a literal value |
| `and` / `or` / `not` | Boolean composition |
| `implies` | If predicate A is true, predicate B must be true |
| `call` | Named predicate resolved by `PredicateRegistry` |

## PredicateRegistry

Most code does not create a registry. `validate(...)` uses wellformed's built-in
predicates automatically:

```ts
import { validate } from "wellformed-ts";

const result = validate(schema, data);
```

Use `PredicateRegistry.withBuiltins()` only when you need to add custom
predicates and keep the built-ins:

```ts
import { PredicateRegistry, createEvalContext, validate } from "wellformed-ts";

const registry = PredicateRegistry.withBuiltins();
registry.register("acme:is_member_id", (value) => {
  return typeof value === "string" && value.startsWith("M-");
});

const context = createEvalContext(registry);
const result = validate(schema, data, { context });
```

`new PredicateRegistry()` starts empty. Use it for isolated tests or specialized
runtimes that intentionally do not want built-ins.

Named predicates use the `call` type and are resolved through the active
registry at runtime.

## Predicate Evolution

A predicate name is a contract. `is_email` can get a faster implementation, but
it should not silently become a different validator.

| Change | How wellformed handles it |
|---|---|
| Faster implementation, same pass/fail answers | Keep the same predicate name |
| Clear bug fix | Fix it, document it in the release notes, and keep a regression test |
| Stricter or looser business rule | Add a new predicate name or explicit `args` |
| External reference data changes | Treat the documented reference source as the contract |
| App-owned predicate changes | Use a namespaced new name or explicit `args` |

Examples:

```json
{ "type": "call", "name": "acme:is_member_id_v2" }
```

```json
{ "type": "call", "name": "is_uuid", "args": { "version": 4 } }
```

The IR does not include a hidden predicate implementation version. That is
intentional. If a schema needs different behavior, the difference should be
visible in the schema.

For audit-sensitive systems, store the application schema id or hash with each
submission. If exact historical behavior matters, also record the
`wellformed-ts` package version or Rust crate version that evaluated it.

Custom predicate implementations are not serialized. Every runtime that
evaluates a schema must register equivalent behavior for each custom predicate
name it uses. Put app predicates under an app namespace, such as
`acme:is_member_id`, to avoid colliding with future built-ins.

## Common Questions

| Question | Answer |
|---|---|
| Can I freeze built-in behavior forever? | Pin the `wellformed-ts` package or Rust crate version. For exact audit replay, store the schema id or hash and the runtime version that evaluated the submission. |
| What if reference data changes? | Built-ins that depend on reference data follow the documented source. If you need a frozen list, model it as an `enum` or an app-owned predicate. |
| Can I override a built-in predicate? | Avoid shadowing built-in names. Use an app namespace such as `acme:is_email` unless you fully own every runtime that will evaluate the schema. |
| Can custom predicates call a database or API? | Keep predicates pure and deterministic. Put network, database, permission, and time-dependent checks in application code outside portable schema validation. |
| How do I know TypeScript and Rust agree? | Test the serialized JSON IR with the same valid and invalid fixtures in every runtime you deploy. |
| Should I store the schema with every submission? | Store at least a schema id or hash. Store the full schema when submissions must be replayable without relying on an external schema registry. |

## Custom Predicates

Register custom predicates for organization-specific validation:

```ts
const registry = PredicateRegistry.withBuiltins();

registry.register("is_even", (value) => {
  return typeof value === "number" && value % 2 === 0;
});

registry.register("divisible_by", (value, args) => {
  if (typeof value !== "number") return false;
  const divisor = (args as { value: number }).value;
  return value % divisor === 0;
});
```

`withBuiltins()` matters here because the registry still needs to know about
built-ins such as `is_email` and `is_uuid` when the same schema uses them.

Use custom predicates through `call` predicates:

```ts
const schema = w.object({
  count: w.integer(),
}).rule(
  { type: "call", name: "divisible_by", args: { value: 5 } },
  "NOT_DIVISIBLE_BY_5",
  "Count must be divisible by 5",
);
```

If `divisible_by` changes meaning later, publish that behavior under a new name
or an explicit `args` option so old schemas keep their original meaning.

## Direct Evaluation

For advanced tooling, you can evaluate predicate IR directly:

```ts
import { createEvalContext, evaluate } from "wellformed-ts";

const ctx = createEvalContext();

evaluate({ type: "regex", pattern: "^\\d+$" }, "123", ctx); // true
evaluate({ type: "range", min: 0, max: 100 }, 50, ctx); // true
evaluate({ type: "exists", path: "/name" }, { name: "Alice" }, ctx); // true
```

Direct evaluation is useful for schema tooling, tests, and custom rule builders. Application validation should usually go through `validate(...)` so transforms, type checks, object rules, and error metadata are handled together.
